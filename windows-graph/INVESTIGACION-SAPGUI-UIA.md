# 1. SAP GUI Scripting

## 1.1 Qué es y por qué existe

Interfaz de automatización COM que expone el árbol de controles de SAP GUI for Windows. **[SAP-DOC]**:

> "From the SAP server's point of view there is no difference between SAP GUI communication generated by a script and SAP GUI communication generated by a user. For this reason a script has the same rights to run SAP transactions and enter data as the user starting it."
> — [SAP GUI Scripting Security Guide, §1](https://help.sap.com/doc/97d2d0bc2ed248a4a85a0bec608704f8/760.01/en-US/sap_gui_scripting_sec_guide.pdf)

**Por qué no puedes usar UIA contra SAP GUI**: SAP documenta que los controles no pueden instanciarse fuera de SAP GUI, así que en lugar de exponerlos por vías estándar de accesibilidad, decidió **exponer una interfaz nueva de automatización** ([SAP Help Portal](https://help.sap.com/docs/sap_gui_for_windows/b47d018c3b9b45e897faf66a6c0885a8/babdf65f4d0a4bd8b40f5ff132cb12fa.html)). En la práctica, los inspectores UIA se atascan en `UIA_PaneControlType` sobre SAP GUI **[COM]**.

→ **UIA sirve para *detectar* la ventana de SAP; Scripting para todo lo demás.** Trátalos como dos backends distintos.

## 1.2 Habilitación: servidor, cliente y el aviso modal

### Servidor **[SAP-DOC]**

> "By default SAP GUI Scripting is disabled on any given SAP system. The administrator has to enable the support by setting the profile parameter `sapgui/user_scripting` to TRUE on the application server."

Se fija con la transacción **RZ11**. Es por *application server*, lo que permite habilitarlo a un subconjunto de usuarios vía logon groups.

Parámetros que modifican el comportamiento **[SAP-DOC]**:

| Parámetro | Efecto | Impacto en TU grabador |
|---|---|---|
| `sapgui/user_scripting` | Habilita todo. Default **FALSE** | Sin esto, nada |
| **`sapgui/user_scripting_disable_recording`** | **"This parameter disables all SAP GUI Scripting events for the system on which it is set."** | ☠️ **Mata tu grabación entera** — ver abajo |
| `sapgui/user_scripting_set_readonly` | Solo lectura de propiedades y funciones read-only | Grabar sí, ejecutar no |
| `sapgui/user_scripting_per_user` | Autoriza por usuario vía objeto de autorización **`S_SCR`**. Default FALSE | Habilitación granular |
| `sapgui/nwbc_scripting` | Equivale a `user_scripting` + `set_readonly` | — |

> ⚠️ **El hallazgo más importante de §1.** `user_scripting_disable_recording` **no desactiva solo el grabador nativo de SAP: desactiva *todos los eventos* de Scripting.** El texto es explícito: *"It is still possible to run previously recorded or written scripts. However, it is not possible to record new scripts or **log any other type of information in response to SAP GUI Scripting events**."*
>
> Es decir: con este parámetro en TRUE, tu `Change`, tu `Hit` y tus handlers **no se disparan nunca**, aunque tu proceso sea externo y no toques el grabador de SAP. Tu ejecución sigue viva; tu grabación no. **Es un interruptor de servidor que un Basis puede activar sin avisarte, y es la causa raíz más probable de "no recibo eventos" en producción.** Detéctalo y degrada explícitamente.

Desde 7.40 PL16 / 7.50 PL3, `set_readonly` y `per_user` **son combinables** (antes `set_readonly` ganaba y anulaba `per_user`) — ver SAP Note 2565390. **[SAP-DOC]**

### Cliente **[SAP-DOC]**

Aunque el admin lo habilite, el usuario puede desactivarlo:

```
HKEY_CURRENT_USER\SOFTWARE\SAP\SAPGUI Front\SAP Frontend Server\Security
    UserScripting     = 0 | 1   (default 1)
    WarnOnAttach      = 0 | 1   (default 1)  <-- el modal
    WarnOnConnection  = 0 | 1   (default 1)
```

> "a user may want to be notified whenever a script attempts to access the running SAP GUI or whenever a script attempts to open a connection to an SAP system. This can be done by setting the registry values `WarnOnAttach` and/or `WarnOnConnection` ... to 1, **which is the default**. Setting these values to 0 will prevent the message popup."

**El warning modal es `WarnOnAttach`, y está ON por defecto.** Se dispara cuando tu proceso se engancha al SAP GUI en ejecución — es decir, en `GetScriptingEngine`. Configurable también desde SAP GUI Options (Accessibility & Scripting → Scripting).

Nota SAP relevante para tu caso de uso **[SAP-DOC]**: *"Since in Accessibility scenarios SAP GUI Scripting is used, it is recommended that users disable the notification regarding attaching scripts to SAP GUI before starting SAP GUI."*

⚠️ **No puedes suprimir el aviso desde tu código de forma fiable.** Existe además `sapgui/user_scripting_force_notification` que, si el admin lo activa, **fuerza la notificación ignorando los ajustes de cliente** **[COM]** — no aparece en la edición 7.60 del Security Guide que verifiqué, trátalo como no confirmado. Escribir `WarnOnAttach=0` en HKCU es una decisión del usuario/despliegue, no algo que tu instalador deba hacer a espaldas del usuario.

**Indicador visual permanente** **[SAP-DOC]**: mientras un script corre, SAP GUI anima un icono en la status bar (temas Blue Crystal y anteriores) o lo muestra en verde en la cabecera (Belize). El tooltip informa si Scripting está desactivado — **es un diagnóstico gratuito**.

**Otros bloqueos**:
- `HKLM\software\sap\SAPGUI Front\SAP Frontend Server\Scripting\DisableWriteToDisk` impide escribir a disco. En SO de 64 bits debe crearse bajo **`Wow6432Node`** **[SAP-DOC]**.
- Si Scripting no se instaló, los User Settings ni aparecen **[SAP-DOC]**.
- El menú *Script Recording and Playback* aparece **greyed out** si Scripting no está habilitado ([KBA 2590231](https://userapps.support.sap.com/sap/support/knowledge/en/2590231)) **[SAP-KB]**.

## 1.3 Árbol de objetos y acceso desde C#

```
GetScriptingEngine  ->  GuiApplication      (/app)
                          .Connections      ->  GuiConnection   (con[0])
                            .Sessions       ->  GuiSession      (ses[0])
                              .ActiveWindow ->  GuiMainWindow   (wnd[0])
                                                  /usr, /mbar, /tbar[0], /sbar ...
```

**[SAP-DOC]** sobre `GuiSession` — define exactamente tu caso de uso:

> "The GuiSession provides the context in which a user performs a certain task such as working with a transaction. It is therefore **the access point for applications which record a user's actions** regarding a specific task or play back those actions."

Y una nota de ciclo de vida crítica **[SAP-DOC]**:

> "the session remains valid until the corresponding main window has been closed, whereas **buttons, for example, are destroyed during each server communication**."

→ **Nunca caches referencias a controles a través de un round-trip al servidor.** Cachea `GuiSession`; re-resuelve todo lo demás con `FindById` tras cada comunicación. Esto no es opinión: la spec dice que los objetos se destruyen.

### Conexión desde C# vía ROT **[COM]** (mecánica ampliamente usada; el moniker `"SAPGUI"` no lo pude verificar en fuente primaria parseable)

SAP publica un helper porque *"Some scripting languages do not allow accessing an entry in the Running Object Table directly"* ([SAP GUI Scripting ROT Entry Helper](https://help.sap.com/docs/sap_gui_for_windows/b47d018c3b9b45e897faf66a6c0885a8/3c48aec0416442eaa8667bd95a1cd8c8.html)).

```csharp
// Referencias: SapROTWr.dll (CSapROTWrapper) e Interop.SAPFEWSELib (tlbimp sobre sapfewse.ocx)
using SAPFEWSELib;
using SapROTWr;

var rot      = new CSapROTWrapper();
object engine = rot.GetROTEntry("SAPGUI");                       // objeto SAPGUI de la ROT

// GetScriptingEngine se invoca por late binding: no está en la interfaz tipada
object app = engine.GetType().InvokeMember(
    "GetScriptingEngine", BindingFlags.InvokeMethod,
    null, engine, null);

var application = (GuiApplication)app;
var connection  = (GuiConnection)application.Children.ElementAt(0);
var session     = (GuiSession)connection.Children.ElementAt(0);

// A partir de aquí, todo por id
var user = (GuiTextField)session.FindById("wnd[0]/usr/txtRSYST-BNAME");
user.Text = "DEV01";
((GuiFrameWindow)session.FindById("wnd[0]")).SendVKey(0);        // Enter
```

**Alternativa sin `SapROTWr`** **[COM]**: `Marshal.GetActiveObject("SAPGUI")` funciona si la app corre en la misma sesión/bitness; `CSapROTWrapper` existe precisamente para los entornos donde no.

Refs **[COM]**: [SAP Q&A: C# can't get session](https://community.sap.com/t5/technology-q-a/sap-gui-scripting-c-can-t-get-session/qaq-p/614735) · [SapGui.Wrapper](https://github.com/alexrivax/SapGui.Wrapper) · [SAP.GUI.Scripting.net](https://github.com/Xiaoy312/SAP.GUI.Scripting.net) (sin mantenimiento)

## 1.4 Element ID, estabilidad y `FindById`

**[SAP-DOC]**:

> "An object's id is created as a unique textual identifier for the object. This is achieved through a URL-like formatting, starting at the GuiApplication object and drilling down to the respective object. For example, the id of the text field in transaction se38 is `/app/con[0]/ses[0]/wnd[0]/usr/ctxtRS38M-PROGRAMM`."

Construcción: `<prefijo de tipo>` + `<name>` (nombre de campo del screen painter / DDIC) + índice si hace falta unicidad + id del padre como prefijo separado por `/`.

### La estabilidad tiene un problema documentado y una solución documentada **[SAP-DOC]**

> "Using these fully qualified ids has one significant drawback: **the index of both the connection and session become part of the id.** If the id given above is used to search for the text field the search will only be successful if the session to which the text field belongs is again the first session of the first connection.
> This problem can be circumvented using **relative ids**. A relative id does not start with `/`... **it is usually preferable to truncate the first part of the id up to the session index** and work with a relative id, such as `wnd[0]/usr/ctxtRS38M-PROGRAMM`. This relative id is still unique with respect to a given session, but it is independent of the number of sessions or connections currently open."

> 🔑 **Regla de diseño, respaldada por la spec, no por folclore**: **persiste ids relativos desde la sesión** (`wnd[0]/usr/...`), nunca absolutos (`/app/con[0]/ses[0]/...`). Un id absoluto grabado en `con[0]/ses[0]` **falla silenciosamente** si el usuario reproduce con dos conexiones abiertas o en otra sesión. Este es el bug de reproducción más común y es 100 % evitable: recorta el prefijo en el momento de grabar.

`FindById` **[SAP-DOC]**: `findById(id As String, raise As Boolean = true) As GuiComponent`. Lo expone cualquier objeto con `GuiVContainer`/`GuiContainer`. Si el id es absoluto, comprueba si el id del contenedor es prefijo y lo trunca — por eso `session.FindById("/app/con[0]/ses[0]/wnd[0]")` y `session.FindById("wnd[0]")` son equivalentes *mientras no haya varias sesiones*.

**`raise:=false`** devuelve `Nothing` en vez de lanzar → úsalo para sondeo (`FindById(id, false) != null`) en lugar de try/catch, que es caro en interop COM **[INF]**.

**`GuiComboBox` — cómo enumerar opciones** **[SAP-DOC]**. Es el único control con opciones enumerables de forma limpia porque *"GuiComboBox retrieves all possible choices on initialization from the server, so the selection is done solely on the client"*:

```csharp
var cmb = (GuiComboBox)session.FindById("wnd[0]/usr/cmbSOMEFIELD");
foreach (GuiComboBoxEntry e in cmb.Entries)      // key + value, ambos String
    Console.WriteLine($"{e.Key} = {e.Value}");
cmb.Key = "02";                                   // seleccionar por clave (preferido)
```

⚠️ *"A combo box can be configured to send a function code to the server when the selection changes, which will invalidate references to all visible elements below the window level."*
⚠️ `.Required` (RO): si está puesto, la entrada vacía no es seleccionable.
⚠️ **`GuiCTextField` ≠ `GuiComboBox`**: se parecen visualmente pero el CTextField **no tiene `Entries`** — sus opciones viven en un dynpro F4 aparte. No puedes enumerar sus "allowedOptions" sin abrir la ayuda.

**`SendVKey`** **[SAP-DOC]**: `sendVKey(VKey As Integer)` está en **`GuiFrameWindow`**, no en los campos. *"The virtual key VKey is executed on the window. The VKeys are defined in the menu painter."* `0` = Enter. `session.GetVKeyDescription(n)` los traduce a texto legible — útil para renderizar el paso grabado en tu UI.

**`GuiTableControl`** **[SAP-DOC]**: `getAbsoluteRow(index As Long) As GuiTableRow`. Sus `Columns` son `GuiTableColumn`, que *"do not support properties like id or name"*.

**`GuiCtrlGridView`** **[SAP-DOC]**: `SelectedRows` es un **string**, *"a comma separated list of row index numbers or index ranges, such as `"1,2,4-8,10"`"*. `CurrentCellRow` va de 0 a n-1, con **-1 = fila de título**. `CurrentCellColumn` es el nombre de campo DDIC.

## 1.6 Enumerar el árbol de la pantalla activa

```csharp
void Walk(GuiComponent c, int depth)
{
    var v = c as GuiVComponent;                  // no todo es visual
    Console.WriteLine(new string(' ', depth * 2) +
        $"{c.Type,-20} id={c.Id} name={c.Name} " +
        $"text={v?.Text} tip={v?.Tooltip} chg={v?.Changeable}");

    if (c.ContainerType)                          // <- la spec dice: solo entonces hay Children
        foreach (GuiComponent child in ((GuiVContainer)c).Children)
            Walk(child, depth + 1);
}

Walk((GuiComponent)session.ActiveWindow, 0);      // ActiveWindow = atajo documentado
```

`session.ActiveWindow` **[SAP-DOC]**: *"most of the time an application will access the currently activated window of the session, as that is the window with which a user will probably interact. This property is intended as a shortcut to this window."*

**Hit test por coordenadas** **[SAP-DOC]** — para un "picker" tipo inspector:

// findByPosition(x, y, raise) As GuiCollection   -- x,y en coordenadas de PANTALLA
var hit = session.FindByPosition(x, y, false);
// Devuelve 2 strings: [0] = Id del componente,

## 1.7 GRABACIÓN — el modelo de eventos (lo crítico)

### 1.7.1 Diseño del modelo, según SAP **[SAP-DOC]**

> "Outgoing interfaces are offered through which an external application can receive notifications about events occurring within the SAP GUI... A user's interactions are best recorded or logged in the context of one session, therefore the **GuiSession exposes an outgoing interface that allows an application to listen to user interactions**."

Esto es explícito y es buena noticia: **SAP diseñó esta API para que un proceso externo grabe al usuario.** No estás abusando de nada. Contrasta con UIA, donde capturar tecleo es estructuralmente imposible (§2.7).

### 1.7.2 El interruptor: `GuiSession.Record` **[SAP-DOC]**

> **`Property record As Boolean`** — "Setting this property to True enables the recording mode of the session. In this mode changes to elements of the user interface are recorded within SAP GUI and sent to a recording application using the **Change** event described later. **This property is not supported by SAP GUI for Java.**"

**Efectos secundarios documentados del modo record** — afectan al usuario que estás observando:
- *"The F4 help dialog is always displayed as a modal window."*
- *"**Drag & Drop is disabled.**"*

> ⚠️ Grabar **no es pasivo**: cambia el comportamiento de la UI bajo el usuario. Si tu app deja `Record=True` permanentemente, el usuario pierde drag & drop en SAP y no sabrá por qué. **Actívalo solo durante la sesión de grabación y restáuralo en `finally`.**

`Property recordFile As String` **[SAP-DOC]**: atajo — pon un nombre de fichero y activa `record`, y SAP escribe un `.vbs` en `SapWorkDir`. *"This property only accepts simple filenames without path information."* Útil como **oráculo de referencia** para validar tu grabador contra el de SAP **[INF]**.

### 1.7.3 Eventos de `GuiSession` **[SAP-DOC]** — firmas literales

```vb
Event startRequest    (session As GuiSession)
Event endRequest      (session As GuiSession)
Event change          (session As GuiSession, component As GuiComponent, commandArray As Object)
Event contextMenu     (session As GuiSession, component As GuiVComponent)
Event destroy         (session As GuiSession)
Event hit             (session As GuiSession, component As GuiComponent, innerObject As String)
Event error           (session As GuiSession, eventId As Long,
                       desc1 As String, desc2 As String, desc3 As String, desc4 As String)
Event automationFCode (session As GuiSession, fcode As String)
```

#### `Change` — el corazón del grabador

> "In record mode, the session **collects** changes to elements of the user interface and **sends these changes to a listening application whenever server communication is about to start** or if the record mode is turned off. The change events are raised **immediately before the startRequest event**. There is at least one event for every modified element in the recorded session."

> 🔑 **Consecuencia arquitectónica que hay que interiorizar**: `Change` **NO es en tiempo real por pulsación**. Es un **flush por lotes**, disparado por el round-trip al servidor. El usuario puede teclear en 5 campos y no recibirás nada hasta que pulse Enter/un botón; entonces recibirás los 5 `Change` de golpe, seguidos de `startRequest`.
>
> **"Tiempo real" en SAP GUI Scripting significa "granularidad de round-trip", no "granularidad de keystroke".** Si tu UX promete feedback por tecla, esta API no lo soporta. Diséñala como "pasos confirmados", que además es la unidad correcta para reproducir.

**Orden garantizado**: `change*` → `startRequest` → (servidor) → `endRequest`.
→ **`startRequest` es tu delimitador de "fin de paso"**: acumula `Change` y materializa el paso al recibirlo. **[INF]**

**La trampa de datos preexistentes** **[SAP-DOC]** — esto invalida grabaciones en producción:

> "**Only changes made at the SAP GUI level are recorded.** Transactions may preset some of the entry fields with values from parameters stored in the SAP system. If these data are not changed in SAP GUI, they will not be recorded. **This may cause problems during playback of scripts, if the entry fields are preset with different values.** If any of the following techniques is used in a transaction, the user should manually modify all the entries he wants to see recorded:
> • Usage of SAP parameters • Variants • Hold Data, from the menu System → User Profile"

> ⚠️ **Un campo correcto porque venía precargado no se graba.** En otra máquina/usuario con otros SET/GET parameters, el replay entra con un valor distinto y **no falla — hace lo incorrecto en silencio**. Mitigación **[INF]**: al cerrar cada paso, haz un barrido del dynpro y persiste el valor *observado* de todo campo `Changeable`, no solo lo que llegó por `Change`. Luego, en replay, **asérta** esos valores en lugar de asumirlos.

Y **[SAP-DOC]**: *"Playback of the changes will only work, if the order of the calls is the same as during recording."* → **preserva el orden; no reordenes ni paralelices pasos.**

#### Formato de `commandArray` **[SAP-DOC]**

> "Each event represents **one line of script code**. The Component parameter specifies the object on which to invoke a method or property. Therefore **the first thing to record is `Component.id`** for later use with `findById`."

`commandArray` = filas de `(Type, Method/Property name, Parameters)`, donde `Type` es `"SP"` (set property) o `"M"` (method):

| Type | Name | Parameters | Significado |
|---|---|---|---|
| `"SP"` | `"Text"` | `"Hello World"` | `.Text = "Hello World"` |
| `"SP"` | `"RecordMode"` | `True` | booleano — *"It is up to the recorder to generate a script line with a valid textual representation"* |
| `"M"` | `"Resize"` | `96, 32, False` | método con 3 args (el 3.º miembro es un array) |

> "There will **usually** be only one line in the array." — pero **no siempre**. Seleccionar una fila de un table control genera **dos**:
>
> | Type | Name | Params |
> |---|---|---|
> | `"M"` | `"getAbsoluteRow"` | `1` |
> | `"SP"` | `"selected"` | `True` |
>
> que se traduce a `session.findById("wnd[0]/usr/tblSAPMBIBSTC537").getAbsoluteRow("1").selected = "True"` — es decir, **el `commandArray` es una cadena de invocación encadenada, no una lista de operaciones independientes.** Tu parser debe componerlas, no ejecutarlas por separado.

**El grabador puede ser más listo que las coordenadas** **[SAP-DOC]** — consejo explícito de SAP:

> "The recorder may however also decide to record other properties of Component. If, for example, a line in a table control or list is selected, **it may be prudent not to record the position of the line, but rather the values in it.** That way, a script can be generated that is more robust with respect to changes in the number, and therefore in the position, of lines. If new function modules have been added, selecting a line from the list might return the wrong function module."

→ SAP te está diciendo que grabes **selección por valor, no por índice**. Aplícalo.

#### `Hit` — hit test, no clics

> "The hit event is only raised **when `elementVisualizationMode` is set to True**, which turns on the hit test mode of SAP GUI. If in this mode a SAP GUI component is identified, the hit event is raised."

`Property elementVisualizationMode As Boolean` (en `GuiApplication`) **[SAP-DOC]**: *"When elementVisualizationMode is enabled, a hit test can be performed on SAP GUI **by moving the cursor over the window**. The hit event of the session is fired when a component was found at the mouse position."*

> ⚠️ **`Hit` NO es un evento de clic.** Es hover/hit-test, y **solo existe bajo `elementVisualizationMode`**. No lo uses para grabar interacciones — para eso está `Change`.
> Su uso correcto: **el modo "inspeccionar/elegir elemento" de tu WPF**, combinado con `GuiVComponent.Visualize(true)` para el marco rojo y `FindByPosition` como alternativa puntual. Y ojo: `elementVisualizationMode` dispara `Hit` **con cada movimiento de ratón** → es una manguera de eventos COM. Actívalo solo mientras el picker esté abierto **[INF]**.

#### Los demás

- **`ContextMenu`** — limitaciones documentadas: *"Only context menus of controls of type GuiShell are supported"* y *"The event is not fired for 'cached' context menus, which are not retrieved from the server when being opened."*
- **`StartRequest`** — *"raised before the session is locked during server communication. At this point user input can be checked before it is sent to the server. **It is not possible to prevent server communication from this event.**"* → es observación, no un hook de veto. No intentes cancelar.
- **`EndRequest`** — *"called immediately after the session is unlocked"* → tu señal de "re-resuelve el árbol; los ids viejos están muertos".
- **`Destroy`** — antes de destruir la sesión → limpia tus sinks.
- **`Error`** — ⚠️ **mucho más estrecho de lo que sugiere el nombre**: *"An error event is currently only raised, if the wrapper library required to access a SAP GUI ActiveX control from a script is not available."* **No es un canal de errores de negocio ni de ABAP.** Para eso lee `GuiStatusbar`. *"error events from all sessions are also available at the GuiApplication."*
- **`AutomationFCode`** — solo con SAP Workplace. Irrelevante hoy **[INF]**.

### 1.7.4 Eventos de `GuiApplication` **[SAP-DOC]**

```vb
Event createSession  (newSession As GuiSession)
Event destroySession (session As GuiSession)
Event error          (eventId As Long, desc1 As String, desc2 As String, desc3 As String, desc4 As String)
```

> `createSession`: "This event is raised whenever a new session is created, **irrespective of whether the session is being created manually, from ABAP or by a script**."
> `destroySession`: "raised before a session is destroyed. This can be done either by closing the main window manually, or by calling the closeSession method."

→ **Este es tu punto de enganche de nivel superior.** Suscríbete a `GuiApplication` una vez y usa `CreateSession` para adjuntar sinks a cada `GuiSession` nueva. Sin esto, un `/o` del usuario abre una sesión que tu grabador no ve. **[INF]**

`Property historyEnabled As Boolean` **[SAP-DOC]**: *"Disabling it will significantly improve the performance of SAP GUI"* — desactiva el autocompletado local. Útil en replay **[INF]**.

### 1.7.6 STA, apartments y el pump — **la trampa de ingeniería nº1**

El encargo dice "Requisito de STA para eventos COM". **Matizo: no es un requisito documentado por SAP** — la spec no menciona apartments. Es una consecuencia del modelo COM. **[COM]/[INF]**

Lo que sí es cierto y verificable:

1. **`sapfewse.ocx` es un servidor COM y sus eventos son connection points.** Un sink en un **STA** solo recibe llamadas cuando ese hilo **bombea mensajes**. Un `Thread` de .NET marcado `[STAThread]` **sin `Dispatcher.Run()`/message loop no recibirá jamás un evento** — el síntoma clásico "me suscribo y no pasa nada".
2. En **WPF esto sale casi gratis**: el hilo de UI ya es STA y ya bombea. Suscribirte desde el hilo de UI de WPF **funciona** para SAP.
3. ⚠️ **Pero eso choca de frente con UIA (§2.6), que Microsoft documenta que debe ir en MTA y fuera del hilo de UI.**

> 🔑 **SAP Scripting y UIA tienen requisitos de threading opuestos e irreconciliables en un mismo hilo.**
> - SAP (COM STA + connection points) → **necesita** un hilo STA **con bomba de mensajes**.
> - UIA → Microsoft **prohíbe** el hilo de UI y **exige MTA** (§2.6, citado literal).
>
> **No los mezcles.** Arquitectura mínima: hilo de UI WPF (STA, bombea) para los sinks de SAP — o un STA dedicado con su propio `Dispatcher.Run()` — **y un hilo MTA separado, sin ventanas, para todo UIA.** Tres hilos, no uno.

**En C#, suscribirse con la interop tipada:**

```csharp
// tlbimp genera delegados <Interfaz>_<Evento>EventHandler
session.Change += new GuiSession_ChangeEventHandler(OnChange);
application.CreateSession += new GuiApplication_CreateSessionEventHandler(OnCreateSession);

void OnChange(GuiSession s, GuiComponent c, object commandArray)
{
    // c.Id ya es válido aquí; el objeto NO lo será tras startRequest.
    // Copia lo que necesites AHORA a un DTO propio.
    var step = new RecordedStep {
        RelativeId = ToRelativeId(c.Id),      // recorta /app/con[i]/ses[j]/  <- §1.4
        Type       = c.Type,
        Commands   = ParseCommandArray(commandArray)
    };
    _pending.Add(step);                        // materializar en startRequest
}
```

> ⚠️ **Mantén vivas las referencias a los delegados** en un campo. Si el sink es recolectado, los eventos dejan de llegar sin error **[COM]** — mismo fallo de raíz que el `GCHandle` de `SetWinEventHook` (§2.7).

## 1.8 Gotchas de SAP

- **`user_scripting_disable_recording` mata todos los eventos** (§1.2). Verifícalo al arrancar y avisa; es indistinguible de un bug tuyo.
- **Scripting deshabilitado por política**: `sapgui/user_scripting=FALSE` (default), `UserScripting=0` en HKCU, o Scripting no instalado. **Diagnóstico gratis**: el tooltip del icono de status bar dice por qué **[SAP-DOC]**; y el menú *Script Recording and Playback* sale greyed out **[SAP-KB]**.
- **`busy`** **[SAP-DOC]**: *"While SAP GUI is waiting for data from the server, **any Scripting call will not return, which blocks the executing thread.** This may not be acceptable for advanced applications. A way to prevent this is to check the busy property of the session."*
  → ☠️ **Si haces esto desde el hilo de UI de WPF, congelas tu app.** Y no puedes simplemente "moverlo a otro hilo" porque los sinks de SAP necesitan el STA que bombea. Comprueba `session.Busy` **antes de cada llamada** y difiere si está TRUE.
- **32/64-bit**: la app y SAP GUI deben poder hablarse. Desde **7.50 PL8 / 7.60**, la instalación escribe automáticamente los valores de registro que permiten usar **`saprotwr.dll` desde procesos de 64 bits** **[COM]**. En versiones anteriores, compila tu WPF como **x86**. Recuerda además `Wow6432Node` para `DisableWriteToDisk` **[SAP-DOC]**.
- **Rotura entre versiones**: el código C# de ROT **se rompió en 7.40→7.70 y otra vez en 7.70→8.0** ([hilo SAP](https://community.sap.com/t5/application-development-and-automation-discussions/sap-gui-scripting-api-c-broken-after-update-7-40-gt-7-70/td-p/12650720)) **[COM]**. Presupuesta pruebas por versión.
- **Un solo diálogo de Recording**: *"The Script Recording and Playback dialog can only be started once"* **[COM]**.
- **`testToolMode`** **[SAP-DOC]**: cambia mensajes I/A de popup a status bar, pone el update en modo inmediato e **ignora mensajes de sistema "so that they do not interrupt the recording or playback of scripts"**. Valores: 0/1. Considéralo para replay desatendido.
- **`GuiApplication` es singleton por proceso** **[COM]**: *"there should always be only one GuiApplication object within a process"*.
- **Nunca escribas contraseñas en scripts** **[SAP-DOC]**: *"Scripts are not encrypted and anybody having access to the file can read the login data."* Como `GuiPasswordField.Text` es solo-escritura, **la API ya te impide grabar contraseñas — no la eludas.**

## 2.2 Árbol, TreeWalker y condiciones

**Vistas** **[MS-DOC]** ([UI Automation Tree Overview](https://learn.microsoft.com/en-us/dotnet/framework/ui-automation/ui-automation-tree-overview)): **Raw** (*"the full tree... the most detailed view available"*), **Control** (*"all UI items from the raw view that an end user would understand as interactive or contributing to the logical structure"*), **Content** (*"UI items that convey the true information"*).

`TreeWalker` **[MS-DOC]** — advertencia literal:
> "Navigating the UI Automation tree using TreeWalker **can result in cross-process calls and is not as efficient** as locating an element using the FindAll or FindFirst methods."

**`TreeWalker.Normalize`** — relevante para tu grabador **[MS-DOC]**: *"Retrieves the node itself, if it satisfies the Condition, or the nearest parent or ancestor node that satisfies the Condition."* Los eventos de foco pueden entregarte elementos **fuera** de tu vista (el ejemplo de los docs: una scrollbar toma el foco pero no es content). `Normalize` los reencaja.

**Condiciones** **[MS-DOC]**:
> ⚠️ **[CORRECCIÓN]** No existe la clase `TrueCondition`. `Condition` es abstracta con **campos estáticos** `Condition.TrueCondition` / `Condition.FalseCondition`. Solo cuatro derivadas: `AndCondition`, `NotCondition`, `OrCondition`, `PropertyCondition`.

En COM: `CreatePropertyCondition(Ex)`, `CreateAndCondition(FromArray|FromNativeArray)`, `CreateOrCondition`, `CreateNotCondition`, `CreateTrueCondition`, `CreateCacheRequest`, `CreateTreeWalker`. **Las variantes `FromArray` importan**: el managed usa `params`, COM necesita array explícito para >2.

**`TreeScope`** **[MS-DOC]**: `Element`=1, `Children`=2, `Descendants`=4, `Subtree`=7, **`Parent`=8 → "Not supported"**, **`Ancestors`=16 → "Not supported"**. Es `[Flags]`, así que una combinación inválida **no falla en compilación**.

**El aviso del RootElement** — `Caution` formal en [Obtaining UI Automation Elements](https://learn.microsoft.com/en-us/dotnet/framework/ui-automation/obtaining-ui-automation-elements) **[MS-DOC]**:
> "In general, you should try to obtain only direct children of the RootElement. A search for descendants may iterate through hundreds or even thousands of elements, **possibly resulting in a stack overflow.**"

⚠️ **Matiz**: avisa sobre **el escritorio**, no en general. *"`TreeScope.Descendants` siempre es lento"* es **[COM]**, no documentado.

## 2.3 Caching — obligatorio para un grabador

**El "por qué"** **[MS-DOC]** — ojo, la mejor cita **no** está en las páginas de caching, sino en [UI Automation Clients Overview](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-clientsoverview):

> "In a single interprocess call, UI Automation retrieves the element and the specified properties and control patterns, and then stores them in the cache. **Without caching, a separate interprocess call is required to retrieve each property or control pattern.**"

Y [Caching UI Automation Properties and Control Patterns](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-cachingforclients):
> "...this method is slow and inefficient because **it requires a cross-process call for each property being retrieved.**"

> ⚠️ **[CORRECCIÓN]** La página de [`Current`](https://learn.microsoft.com/en-us/dotnet/api/system.windows.automation.automationelement.current) **no dice nada** sobre coste cross-process. Cita las páginas conceptuales, no `Current`.

**Matiz que te afecta** **[MS-DOC]**: *"The benefits of caching are most noticeable with WPF controls and custom controls that have server-side UI Automation providers. **There is less benefit when accessing client-side providers such as the default providers for Win32 controls.**"*

**Gotchas de `CacheRequest`** **[MS-DOC]**:
- *"When a pattern is cached, its properties are **not** automatically cached"* — añádelas con `CacheRequest.Add`.
- *"if you set only Children... the properties and patterns of children are cached, **but not those of the element itself**. To ensure that caching is done for the retrieved element itself, you must include `Element`."*
- *"**By default, caching is performed only for elements that appear in the control view.**"* → pon `TreeFilter = Automation.RawViewCondition` para ampliar. **Descarta elementos raw en silencio.**
- *"The cache is valid only as long as nothing changes in the UI."* Refresca con `GetUpdatedCache` / COM `BuildUpdatedCache`.
- **Tu palanca clave**: *"If you subscribe to an event while a CacheRequest is active, you obtain an AutomationElement with an updated cache as the source of the event."* Pero *"Any changes made to the CacheRequest after you subscribe to the event have no effect."*
  → **Suscríbete con un `CacheRequest` activo y los `sender` de los eventos llegan pre-poblados** — exactamente lo que un grabador necesita, y evita el round-trip en el callback.

> ⚠️ **[CORRECCIÓN]** No existen `FindFirstBuildCache`/`FindAllBuildCache` en managed (son COM-only). El caching managed es **ambiental**: activas un `CacheRequest` y llamas a los `FindFirst`/`FindAll` normales. **`TreeWalker` es la excepción** — *"caching is only done if a CacheRequest is specified as a parameter."*

**`AutomationElementMode`** — la restricción arquitectónica más importante **[MS-DOC]**:
> "Certain operations on elements, such as GetCurrentPropertyValue or SetFocus, require a full reference; attempting to perform these on an element that has none results in an **InvalidOperationException**."

> 🔑 **Tus dos mitades tienen necesidades opuestas.** El **grabador** puede usar `AutomationElementMode.None` (snapshots baratos, sin referencia viva). El **ejecutor debe usar `Full`**, o `Invoke`/`SetFocus` y todo patrón de acción quedan inutilizables. **No pongas `None` globalmente.**

## 2.4 Identificación estable

**`AutomationId`** **[MS-DOC]** ([propids](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-automation-element-propids), `UIA_AutomationIdPropertyId`=30011):
> "the AutomationId of an element must be the same in any instance of the application, **regardless of the local language**. **The value should be unique among sibling elements, but not necessarily unique across the entire desktop**... this property **is not mandatory**... **AutomationId is not guaranteed to be stable across different releases or builds of an application.**"

De [Use the AutomationID Property](https://learn.microsoft.com/en-us/dotnet/framework/ui-automation/use-the-automationid-property):
> "AutomationIdProperty does not guarantee a unique identity throughout the tree; **it typically needs container and scope information to be useful.**"

**Microsoft documenta el path cualificado por ancestros** — cita esto, no folclore:
> "...since AutomationID is only guaranteed to be unique amongst siblings, multiple elements in the UI Automation tree may have identical AutomationID property values. **In these situations the elements can be uniquely identified based on a parent and, if necessary, a grandparent.**"

**Y un aviso apuntado directamente a tu caso de uso** **[MS-DOC]**:
> "**A major drawback to using AutomationID for recording user interactions in a volatile UI is the probability of catastrophic change in the UI.**"

**Ausente en** (literal): *"top-level application windows, UI Automation elements derived from WPF controls that do not have an ID or x:Uid, and UI Automation elements derived from Win32 controls that do not have a control ID."* Default: string vacío.

**`Name`** — la cita para "no construyas selectores sobre Name" **[MS-DOC]**:
> "**The Name property cannot be used as a unique identifier among siblings.** ... **For test automation, the clients should consider using the AutomationId or RuntimeId property.**"

Además: Edit — *"The Name property should never contain the textual contents of the edit control"*; ComboBox — *"should never contain the current contents of the combo box"*. **`Name` está localizado; `AutomationId` no.**

**`ClassName`** (30012) **[MS-DOC]**: *"is not always in a standard format. However, if the class name is known, it can be used to **verify** that an application is working with the expected automation element."* → **documentado para verificar, no para identificar.**
**`FrameworkId`** (30024): *"Win32", "WinForm", "DirectUI"*.

**`RuntimeId`**:
> ⚠️ **[CORRECCIÓN]** El encargo pedía citarlo como "NOT persistent across sessions/runs". **Esa frase no existe** en ninguna de las tres páginas. Lo documentado es:
> "The identifier is only guaranteed to be unique to the UI of the desktop on which it was generated. **Identifiers can be reused over time.** The format of run-time identifiers might change in future releases. **The returned identifier should be treated as an opaque value and used only for comparison.**"
>
> **Formulación defensible**: cuatro hechos — ámbito de escritorio, **reutilización en el tiempo**, formato cambiante, opaco/solo-comparación. **Reutilización + opacidad es lo que justifica "nunca lo persistas"** (un RuntimeId guardado puede colisionar luego con **otro** elemento) — pero eso es **[INF]** tuya, no una frase de Microsoft. Microsoft lo posiciona como clave de identidad **intra-sesión**.

### Estrategia de selector **[COM]** (consistente con las citas de arriba, pero la receta es de la comunidad)

```
1. AutomationId                        (si != "")
2. AutomationId + cadena de ancestros  (Microsoft lo documenta como la mitigación)
3. ControlType + Name                  (⚠️ Name está localizado)
4. + ClassName como desempate          (documentado solo para verificar)
5. Path por índices ordinales          (último recurso: cualquier hermano insertado lo rompe)
```

## 2.5 Patterns

**`ValuePattern`** **[MS-DOC]** ([Implementing Value](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-implementingvalue)): `IsReadOnly`, `Value`, `SetValue`. **"This control pattern has no associated events."**
Excepciones: read-only → `InvalidOperationException`; deshabilitado → `ElementNotEnabledException`.

> ⚠️ **La trampa multilínea, literal**: *"Single-line edit controls support programmatic access to their contents through ValuePattern. However, **multi-line edit controls do not support the ValuePattern**; instead they provide access to their content through the TextPattern."* Y el propio sample de Microsoft: *"TextPattern does not support setting the text... For this reason, **text input must be simulated**."*
> → **En un edit multilínea no puedes `SetValue`. Tienes que sintetizar teclado.**

**Passwords** **[MS-DOC]**: *"If the `UIA_IsPasswordPropertyId` property is set to TRUE, querying the Value property **must return an error**."* (Simetría agradable con `GuiPasswordField` de SAP.)

**`LegacyIAccessiblePattern`** — **COM-only, verificado** **[MS-DOC]** ([pattern](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-implementinglegacyiaccessible)):
> "The LegacyIAccessible control pattern is supported by the **Microsoft Active Accessibility to Microsoft UI Automation Proxy**... enabling them to access the underlying IAccessible implementation."
> "**Application and control providers never implement the ILegacyIAccessibleProvider interface.**"
> "**The LegacyIAccessible control pattern is not available for controls based on UI Automation.**"

Miembros: ChildId, DefaultAction, Description, Help, KeyboardShortcut, Name, Role, State, **Value**; `DoDefaultAction`, `GetIAccessible`, `GetSelection`, `Select`, **`SetValue`**.
→ **Tu fallback cuando falta `ValuePattern`. Solo alcanzable desde COM. Este es el motivo real de elegir UIA3.**

**`Selection` / `SelectionItem`** **[MS-DOC]**: `CanSelectMultiple`, `IsSelectionRequired`, `GetSelection`. *"Menus do not support the Selection control pattern."*
No caches estos dos: *"The `IsSelectionRequired` and `CanSelectMultiple` properties **can be dynamic**."*

**`TogglePattern`** **[MS-DOC]** — dos trampas:
> "A control must **cycle** through its toggle states in the following order: On, Off and, if supported, Indeterminate."
> "**Toggle does not provide a set-state method.**"
> "**The radio button control does not implement IToggleProvider**, because it is not capable of cycling through its valid states."

→ **No hay `SetState`**: llama `Toggle()` en bucle releyendo estado, gestionando el ciclo de 3. **Y los radio buttons NO son toggles** — usa `SelectionItemPattern.Select()`. *Es el bug de automatización más común.*

**`ExpandCollapsePattern`** **[MS-DOC]**: *"ExpandCollapseState refers to the visibility of **immediate child objects only**"*. Sin eventos propios → usa property-changed.

**`TextPattern`**: para Edit, `ITextProvider` es **Required** — *"All edit controls must support the Text control pattern."* **No puede fijar texto.**

### ControlType → Patterns **[MS-DOC]** ([Control Patterns Mapping](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-controlpatternmapping))

| ControlType | Obligatorio | Condicional | NO soporta |
|---|---|---|---|
| **Button** | *ninguno* | ExpandCollapse, Invoke, Toggle, Value | — |
| **CheckBox** | **Toggle** | — | — |
| **ComboBox** | **ExpandCollapse** | Selection, Value | Scroll |
| **Edit** | *ninguno* | RangeValue, Text, Value | — |
| **RadioButton** | **SelectionItem** | — | **Toggle** |
| **List** | *ninguno* | Grid, MultipleView, Scroll, Selection | Table |
| **ListItem** | **SelectionItem** | ExpandCollapse, GridItem, Invoke, ScrollItem, Toggle, Value | — |
| **DataGrid** | **Grid** | Scroll, Selection, Table | — |

- **Button no obliga a nada** → **hay que sondear**, no asumir `Invoke`.
- **List/Tree**: `Selection` es solo *condicional* en el contenedor, pero ListItem **debe** soportar `SelectionItem` → **ir de abajo arriba (`SelectionItemPattern.Select()` en el ítem) es más robusto que de arriba abajo.**
- ⚠️ **Contradicción documental**: la tabla de mapping dice que Text es condicional en Edit; la [página de Edit](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-supporteditcontroltype) dice **Required**. **Fíate de la página del control type.**

**Dónde falta `ValuePattern`** **[MS-DOC]** ([Support for Standard Controls](https://learn.microsoft.com/en-us/dotnet/framework/ui-automation/ui-automation-support-for-standard-controls)): WinForms DataGridView, ToolStrip, MenuStrip, NumericUpDown, MaskedTextBox, LinkLabel, PropertyGrid, Panel, Form, UserControl están expuestos *"only through their support for Microsoft Active Accessibility. **Some functionality may not be available.**"* Además: *"**Full support is provided only for controls from version 6 of ComCtrl32.dll**"*; no soportados: `SysAnimate32`, `SysPager`, `SysDateTimePick32`, `SysMonthCal32`, `ScrollBar` suelto.
→ **Esta tabla es exactamente donde necesitarás el fallback LegacyIAccessible de UIA3.**

### Enumerar las "allowedOptions" de un combo — parcialmente documentado

**[MS-DOC]** ([ComboBox control type](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-supportcomboboxcontroltype)): `ExpandCollapse` **obligatorio** (*"a combo box control must always contain a drop-down button"*); `Selection` *"delegated to the list box beneath the combo box, but **may not always be feasible**"*; `Value` obligatorio *"If the combo box can take arbitrary text values"*.

**La cita clave** — Microsoft sanciona explícitamente la población perezosa ([ExpandCollapse](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-implementingexpandcollapse)):
> "**If descendants are generated on demand, they may only appear in the UI Automation tree after being displayed for the first time or only while they are visible.**"

→ **No hay garantía documentada de poder enumerar sin expandir, ni requisito documentado de expandir. Es dependiente del provider, por diseño.**

**El mecanismo *correcto* y documentado para ítems virtualizados** — que la mayoría de guías ignora:
- [`VirtualizedItemPattern.Realize()`](https://learn.microsoft.com/en-us/dotnet/api/system.windows.automation.virtualizeditempattern) — *"Makes the virtual item fully accessible as a UI Automation element."*
- [`ItemContainerPattern`](https://learn.microsoft.com/en-us/dotnet/api/system.windows.automation.itemcontainerpattern) — recupera un ítem por valor de propiedad **sin realizar toda la lista**.

**[COM]** En la práctica hay que expandir: [FlaUI #544](https://github.com/FlaUI/FlaUI/discussions/544), [#579](https://github.com/FlaUI/FlaUI/issues/579), [#253](https://github.com/FlaUI/FlaUI/issues/253) (carrera con la animación), [MS Q&A](https://learn.microsoft.com/en-us/answers/questions/395831/wpf-ui-automation-how-to-get-combobox-item). [`ComboBox.cs` de FlaUI](https://github.com/FlaUI/FlaUI/blob/master/src/FlaUI.Core/AutomationElements/ComboBox.cs) llama a `Expand()` antes de leer `Items`, espera `AnimationDuration = 100ms`, y contiene un hack `Expand(); Collapse();` comentado *"UIA2/WinForms does not set the selected item until it is expanded."*

**Receta**: `ItemContainerPattern`/`Realize()` si el provider los soporta; si no, expandir-primero + espera de asentamiento.

## 2.7 Lo que UIA **no** puede hacer

### Capturar tecleo: estructuralmente imposible

**Ninguna API de UIA reporta pulsaciones a un observador pasivo.** El único pattern de input, [SynchronizedInput](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-implementingsynchronizedinput), hace lo opuesto **[MS-DOC]**: *"enables client applications to **direct** the mouse or keyboard input to a specific UI element"* — es verificación para *replay*, no captura, y *"The UI Automation provider must **discard** the input if it is for an element other than the current element."*

**`TextChangedEvent` no garantiza casi nada.** Su definición completa es *"Identifies the event raised whenever textual content is modified"*, y **el tipo de args es `AutomationEventArgs` plano — sin payload**. Y **[MS-DOC]** ([uiauto-eventsoverview](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-eventsoverview)):
> "Some events do not necessarily mean that the state of the UI has changed... a **UIA_Text_TextChangedEventId event is raised, even if the user did not actually change the text.** When processing an event, it may be necessary for a client application to check whether anything has actually changed."

→ **TextChanged es una bandera de "sucio".** Al recibirlo, relee `ValuePattern.Current.Value`. **No puedes reconstruir keystrokes.**

### ¿Dispara `ValueProperty` en edits Win32? Los docs se contradicen

**El requisito** **[MS-DOC]** ([Edit control type](https://learn.microsoft.com/en-us/windows/win32/winauto/uiauto-supporteditcontroltype)): *"UIA_ValueValuePropertyId property-changed event | If the control supports the Value control pattern, **it must support this event**"* — pero acotado a *"where the UI framework/platform integrates UI Automation support"*. **Los controles Win32 legacy se exponen por *proxies*, no por soporte integrado.**

**La escapatoria de Microsoft — la frase más importante para tu análisis de riesgo.** Está en las páginas de eventos Win32 **y** .NET **[MS-DOC]**:
> "Do not assume that all possible events are raised by a UI Automation provider. For example, **not all property changes cause events to be raised by the standard proxy providers for Windows Forms and Microsoft Win32 controls.**"

> 🔑 **El requisito es una obligación para los providers, no una promesa a los clientes.** Las dos frases están en tensión; **para ti opera la segunda.**

**[COM]** Caso confirmado por Microsoft: [dotnet/winforms#7763](https://github.com/dotnet/winforms/issues/7763) — *"ComboBox supports Value UIA pattern, however, does not raise the property change event for value property"* — **cerrado**, etiquetado "MAS violation, UIA issue". Narrator solo funciona escuchando eventos de **selección** en su lugar.

> ⚠️ **[CORRECCIÓN]** No encontré cita alguna para *"ValueProperty solo dispara al perder el foco"* en Edits Win32. Muy repetido en círculos RPA; **sin verificar**. Mídelo contra tus targets reales en vez de diseñar alrededor.

### El fallback: WinEvents **[MS-DOC]** para las APIs, **[COM]** para la práctica

[`SetWinEventHook`](https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-setwineventhook):
> "The client thread that calls SetWinEventHook **must have a message loop** in order to receive events."
> "When you use SetWinEventHook to set a callback in managed code, you should use the **GCHandle** structure to avoid exceptions. This tells the garbage collector not to move the callback."

⚠️ El `GCHandle` es un footgun real de C#: un delegado no enraizado se recolecta y **crashea**.

> 🔑 **WinEvents invierten el modelo de threading de UIA**: WinEvents **exigen** bomba de mensajes y entregan en el hilo llamante; UIA **prohíbe** el hilo de UI y entrega en el suyo. **No pueden compartir hilo.**

[Event Constants](https://learn.microsoft.com/en-us/windows/win32/winauto/event-constants):
- **`EVENT_OBJECT_VALUECHANGE` (0x800E)** — *"**The system sends this event** for the user interface elements that include the scroll bar and the following controls: **edit**, header, hot key, progress bar, slider, and up-down."*
  → **Lo envía el propio sistema para edits — por eso es el fallback exacto cuando el proxy de UIA se calla.**
- `EVENT_OBJECT_FOCUS` (0x8005), `EVENT_OBJECT_SELECTION` (0x8006)
- `EVENT_OBJECT_INVOKED` (0x8013) — *"supported by common controls and is **used by UI Automation**"*

Microsoft recomienda verificar con [Accessible Event Watcher](https://learn.microsoft.com/en-us/windows/win32/winauto/accessible-event-watcher) antes de depender de ellos.

**Hook de teclado de bajo nivel** — **aquí sí vive la regla documentada de "vuelve rápido"** ([`LowLevelKeyboardProc`](https://learn.microsoft.com/en-us/windows/win32/winmsg/lowlevelkeyboardproc)) **[MS-DOC]**:
> "...on Windows 7 and later, **the hook is silently removed without being called. There is no way for the application to know whether the hook is removed.**"
> "**Windows 10 version 1709 and later** The maximum timeout value the system allows is 1000 milliseconds."
> "If the application must use low level hooks, it should run the hooks on a **dedicated thread that passes the work off to a worker thread and then immediately returns.** In most cases where the application needs to use low level hooks, it should monitor **raw input** instead."

⚠️ **Un hook eliminado en silencio sin notificación significa que tu grabación simplemente para.** Microsoft recomienda [Raw Input](https://learn.microsoft.com/en-us/windows/desktop/inputdev/raw-input).

## 3.2 Qué es el "selector" en cada superficie

> **La diferencia estructural que gobierna el diseño:**
>
> - **DOM**: el selector es una **consulta** (declarativa, re-evaluada).
> - **SAP**: el selector es una **ruta canónica que el sistema te da**. Es la superficie *más fácil* de las tres — `Id` es único, textual, estable y derivado del DDIC, no de la posición en pantalla. **Todo el problema de identificación se reduce a "recorta el prefijo de conexión/sesión" (§1.4).**
> - **UIA**: **no existe selector nativo.** `AutomationId` es opcional, solo único entre hermanos, y *"not guaranteed to be stable across different releases or builds"*. **Tienes que fabricar el selector** — y Microsoft avisa expresamente de *"a major drawback to using AutomationID for recording user interactions in a volatile UI"*.
>
> → **Presupuesta el esfuerzo asimétricamente: la identificación en SAP es casi gratis; en UIA es el grueso del trabajo.**

## 3.3 Clasificación de `actionType`

| actionType | SAP: cómo lo detectas al grabar | SAP: cómo lo ejecutas | UIA: cómo lo detectas | UIA: cómo lo ejecutas |
|---|---|---|---|---|
| **input** | `Change` con `("SP","Text",v)` sobre `txt`/`ctxt`/`pwd` | `.Text = v` | `PropertyChanged(ValueProperty)` (⚠️ no fiable en Win32 → `EVENT_OBJECT_VALUECHANGE`) | `ValuePattern.SetValue` (⚠️ **no en multilínea** → sintetizar teclado) |
| **select** | `Change` con `("SP","Key"\|"Value",v)` sobre `cmb`; o `("SP","Selected",True)` sobre `rad` | `.Key = v` / `.Select()` | `ElementSelectedEvent` | `SelectionItemPattern.Select()` |
| **toggle** | `Change` con `("SP","Selected",b)` sobre `chk` | `.Selected = b` | `PropertyChanged(ToggleStateProperty)` | `TogglePattern.Toggle()` **en bucle** (no hay SetState) |
| **click** | `Change` sobre `btn` + `startRequest` | `.Press()` | `InvokedEvent` | `InvokePattern.Invoke()` |
| **navigation** | `startRequest`/`endRequest`; `Type=="GuiOkCodeField"`; `GuiApplication.CreateSession` | `SendCommand`/`StartTransaction` | `WindowOpened`/`WindowClosed` | (implícito) |
| **grid/table** | `Change` **multi-línea** encadenada (`getAbsoluteRow` + `selected`) | `GetAbsoluteRow(i)` / `ModifyCell(r,c,v)` | `ElementSelected` en el `DataItem` | `SelectionItemPattern.Select()` en el ítem (**abajo-arriba**) |

**Heurística de clasificación en SAP** **[INF]**, derivada de la spec: clasifica por **`component.Type`** (documentado como discriminador fiable) **y** por la forma del `commandArray` — `"M"` + `"press"` → click; `"SP"` + `"Text"` → input; `"SP"` + `"Key"` → select. **No infieras del valor.**

**Heurística en UIA** **[INF]**: clasifica por el **evento**, no por el ControlType (que promete poco: Button no obliga a ningún pattern). `Invoked` → click; `ElementSelected` → select; `PropertyChanged(Value)` → input; `PropertyChanged(ToggleState)` → toggle.

# 4. Correcciones a las premisas del encargo

Cuatro afirmaciones del brief no se sostienen. Están corregidas en su sección; se recopilan aquí porque afectan al diseño:

1. **§2.1 — "Microsoft recomienda la COM por rendimiento/cuelgues."** No existe tal declaración. El blog del equipo de UIA dice *"There's nothing wrong with using the wrapper classes"*; la razón documentada para COM es **desfase de features**. La frase "improved reliability and performance" es de **TestStack**. **Argumenta UIA3 desde `LegacyIAccessiblePattern` (COM-only, verificado) + desfase de features.**
2. **§2.6 — "No hacer trabajo pesado ni llamar UIA en el hilo del callback."** Ninguna de las cinco páginas candidatas lo dice. Y Microsoft documenta **lo contrario** sobre la segunda mitad: *"It is safe to make UI Automation calls in a UI Automation event handler."* La regla "vuelve rápido" **sí existe, pero para `WH_KEYBOARD_LL`**. Justifica tu cola con *"the event handler is always called on a non-UI thread"* (→ debes marshalear al Dispatcher igualmente).
3. **§2.4 — RuntimeId "no persistente entre sesiones".** No documentado así. Documentado: ámbito de escritorio, **"Identifiers can be reused over time"**, formato cambiante, opaco/solo-comparación. "Nunca lo persistas" es inferencia legítima **desde reutilización + opacidad**, no una frase de Microsoft.
4. **§1.7.6 — "Requisito de STA para eventos COM" (SAP).** **SAP no documenta apartments.** Es consecuencia del modelo COM, no un requisito de la API. Lo importante es lo que se deriva: **STA sin bomba de mensajes no recibe eventos**, y **eso choca con el MTA obligatorio de UIA.**

Menores: no existe la clase `TrueCondition` (es campo estático de `Condition`); no hay `FindFirstBuildCache`/`FindAllBuildCache` en managed (COM-only); el orden de `AddAutomationEventHandler` es `(eventId, element, scope, handler)`; *"`TreeScope.Descendants` siempre es lento"* es comunidad — los docs solo avisan sobre el **escritorio**.

# 5. Arquitectura que se deduce de los hechos

```
┌─ Hilo UI WPF (STA, con bomba de mensajes) ──────────────────────┐
│  Dispatcher · ViewModels                                        │
│  Sinks de SAP GUI Scripting  <-- necesitan STA + pump           │
│    GuiApplication.CreateSession -> adjuntar sink por sesión     │
│    GuiSession.Change  (acumular)                                │
│    GuiSession.StartRequest (materializar paso)                  │
│  ⚠️ Comprobar session.Busy ANTES de cada llamada, o congelas    │
└─────────────────────────────────────────────────────────────────┘
┌─ Hilo MTA dedicado (sin ventanas) ──────────────────────────────┐
│  TODO UIA: suscribir, desuscribir y llamar. Documentado.        │
│  Suscribir bajo un CacheRequest activo -> senders pre-poblados  │
└─────────────────────────────────────────────────────────────────┘
┌─ Hilo con bomba de mensajes (separado) ─────────────────────────┐
│  SetWinEventHook (EVENT_OBJECT_VALUECHANGE) — fallback de texto │
│  Delegados enraizados con GCHandle                              │
└─────────────────────────────────────────────────────────────────┘
                          |  marshal
                          v
                  Cola de pasos grabados
```

**Tres hilos, no uno.** Los requisitos son mutuamente excluyentes:

1. **SAP Scripting exige STA + bomba**; **UIA exige MTA + sin ventanas** (documentado, literal); **WinEvents exigen bomba y entregan en el hilo llamante** (documentado). **No hay hilo que satisfaga los tres.**
2. **`session.Busy` bloquea el hilo llamante** *"any Scripting call will not return"* — y ese hilo es tu UI. Comprueba siempre antes.
3. **Grabador puede usar `AutomationElementMode.None`; ejecutor debe usar `Full`** o pierdes `Invoke`/`SetFocus`.
4. **Silencia el grabador durante el replay** — tus propias acciones disparan `ElementSelected`/`Change`.
5. **Re-resuelve todo tras `endRequest`**: los objetos SAP bajo el nivel de ventana *"are destroyed during each server communication"*.
6. **Persiste ids relativos, nunca absolutos** (§1.4).
7. **Graba selección por valor, no por índice** — consejo explícito de SAP.
8. **Barre y persiste los valores de todo campo `Changeable`**, no solo lo que llegó por `Change` — los campos precargados **no se graban** (§1.7.3).

## Dónde falla cada superficie

| | SAP GUI Scripting | UIA |
|---|---|---|
| **Identificación** | ✅ Casi resuelta (`Id` canónico) | ❌ El grueso del trabajo |
| **Granularidad de grabación** | ⚠️ Por round-trip, no por tecla | ⚠️ Por evento, sin tecleo |
| **Captura de texto** | ✅ Vía `Change` (batched) | ❌ **Estructuralmente imposible** → híbrido con WinEvents/Raw Input |
| **Opciones de combo** | ✅ `Entries`, limpio | ⚠️ Dependiente del provider |
| **Interruptor de apagado ajeno** | ☠️ `user_scripting_disable_recording` | — |
| **Fiabilidad de eventos** | ✅ Alta si está habilitado | ⚠️ *"not all property changes cause events... for Win32 controls"* |

> **La conclusión de una línea**: en SAP, la identificación es fácil y la grabación es fiable pero *batched*, y todo depende de un flag de servidor que no controlas. En UIA, los clics y la selección funcionan bien, **pero el texto es donde UIA falla estructuralmente** — ahí necesitas obligatoriamente `EVENT_OBJECT_VALUECHANGE` (que *"the system sends"* para edits) o Raw Input.
